1. Назначение программы
Данная программа представляет собой веб-контейнер верификации обновлений, предназначенный для проверки подлинности и целостности файлов обновлений. Приложение позволяет создавать пользовательские файлы, формировать для них электронную подпись Ed25519, выполнять проверку подписи, вычислять контрольные хэш-значения SHA-256 и SHA-512, формировать итоговый статус верификации и при необходимости перемещать подозрительные файлы в карантин. В проекте также предусмотрен встроенный функционал симуляции работы системы на наборе тестовых файлов с различными состояниями.

2. Общая логика работы
Пользователь с помощью веб-интерфейса создает файл обновления в каталоге пользовательских файлов. Для дальнейшей корректной проверки для этого файла должна существовать соответствующая электронная подпись в каталоге подписей. При запуске проверки контейнер вычисляет SHA-256 и SHA-512 для выбранного файла, считывает открытый ключ из заданного в конфигурации файла, проверяет подпись и присваивает итоговый статус ACCEPTED или REJECTED. Если подпись отсутствует либо недействительна, приложение может предложить переместить файл в карантин.

3. Назначение основных модулей
manage.py
Используется для стандартного запуска команд Django-проекта.

django_project/settings.py
Конфигурационный модуль Django, определяющий базовую директорию проекта и префиксы для доступа к медиафайлам.

django_project/urls.py
URL‑конфигурация проекта.

core/config.py
Содержит основные пути к рабочим каталогам и файлам ключей. Через этот модуль задаются директории пользовательских файлов обновлений, файлов подписей, карантина, а также путь к открытому ключу, ограничения на количество файлов и параметры симуляции.

core/paths.py
Отвечает за подготовку структуры директорий.

core/verifier.py
Реализует вычисление хэш-значений, загрузку открытого ключа, криптографическую проверку подписи, формирование результата верификации и подготовку данных для дальнейшего перемещения отклонённых файлов в карантин.

core/signer.py
Отвечает за формирование электронной подписи для файлов обновлений с использованием закрытого ключа. Для Ed25519 применяется пара ключей, где закрытый ключ используется для подписи, а открытый — для последующей проверки подписи.

core/file_manager.py
Используется для создания, сохранения и базовой обработки пользовательских файлов обновлений. Через него выполняются получение списка файлов, проверка имени, нормализация имени, валидация содержимого и генерация случайного содержимого.

core/utils.py
Содержит вспомогательные функции: построение путей к файлам подписей, перечисление файлов в каталогах.

core/compromise_manager.py
Используется для моделирования нарушений целостности файла или подписи. Через этот модуль можно изменить содержимое файла, удалить подпись либо очистить файл подписи.

core/quarantine_manager.py
Отвечает за перенос файлов и подписей в карантин, отображение содержимого карантина и его очистку.

core/simulation.py
Используется для имитационного тестирования работы контейнера на наборе файлов с различными состояниями: корректная подпись, отсутствие подписи, изменённый файл и другие варианты. По итогам симуляции формируются результаты, сводка и графические материалы.

web_container/views.py
Содержит представления Django, связывающие HTML-страницы и прикладную логику core. Через views пользователь получает доступ к созданию файлов, удалению, подписи, проверке, компрометации, карантину и симуляции.

workflows/
Содержит вспомогательные сценарии подготовки среды, в том числе workflow для генерации пары ключей.

4. Подготовка программы к работе
Перед первым запуском необходимо:

установить зависимости из файла requirements;

сгенерировать пару ключей Ed25519 с помощью Generate Key Pair из workflows;

убедиться, что открытый ключ сохранён в путь, указанный в конфигурации, а закрытый ключ помещен в Secrets под названием, совпадающим с заданным в config.py;

убедиться, что рабочие каталоги для обновлений, подписей и карантина существуют или будут автоматически созданы при запуске;

выполнить стандартную подготовку Django-проекта и запустить сервер.

5. Конфигурация
Основные параметры задаются в модуле core/config.py. В нём определяются:

каталоги пользовательских файлов обновлений;

каталоги файлов подписей;

каталоги карантина;

путь к открытому ключу;

путь к закрытому ключу или способ его получения;

ограничения на количество пользовательских файлов;

параметры симуляции, включая количество файлов, число компрометированных экземпляров и имена диаграмм.

6. Используемые библиотеки
Состав библиотек определяется файлом requirements. Ключевыми зависимостями проекта являются Django для реализации веб-интерфейса и маршрутизации, а также библиотека ed25519 для формирования и проверки электронной подписи файлов.

7. Инструкция по работе пользователя
Общий порядок работы с программой следующий:

запустить проект;

открыть главную страницу веб-приложения;

выбрать нужное действие в интерфейсе;

выполнить требуемую операцию в соответствии с формой или страницей подтверждения;

при необходимости просмотреть результат проверки, карантин или результаты симуляции.

Если после выполнения действия пользователь видит ту же форму повторно, это обычно означает, что приложение вернуло страницу с сообщением об ошибке и сохранило ранее введённые значения для повторного редактирования. Такой механизм применяется, например, при некорректном имени файла, превышении лимита количества файлов или ошибке содержимого.
